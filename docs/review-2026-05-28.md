# tanker-flow project review — 2026-05-28

Snapshot review of the pipeline, the live database state, code-level gaps, and
recommended next steps. Generated against branch `master` at commit
`0793e79` (feat(pipeline): layered laden inference + draught popup).

## TL;DR

The pipeline is well-architected and the state machine is genuinely impressive,
but the **central market signal (`laden ton-miles in transit`) is not yet
computed**, and the live data reveals a **critical gap in zone polygons** that
silently breaks the downstream signal even once it's wired up. Three concrete
next steps below.

---

## 1 · Critical findings (must-fix before signal is meaningful)

### 1.1 Zero terminals have an `approach` polygon — the macro-envelope layer is dead

`CLAUDE.md` and `README.md` describe `approach` as the **macro envelope
containing anchorage + berth** that keeps a visit "open" during channel
transit. `db/init/schema.sql:84-89` documents it. `pipeline/state_machine.py`
lines 27, 199-215 treats `approach` as a first-class polygon. In the QGIS
source, **no `approach` polygons have ever been drawn**:

```
33 in-scope terminals · approach polygon coverage: 0
```

`qgis/README.md` confirms this — its layer schema only allows `berth |
anchorage`. So **layer 3 of the state machine's resolution is dead code**, and
the consequences in the actual `port_events` table are stark:

| event_type | count |
|---|---|
| zone_entry | 401 |
| zone_exit | 385 |
| moored | **181** |
| anchorage_entry | 111 |
| anchorage_exit | 105 |
| anchored | 62 |
| **`departed`** | **0** |

**Zero `departed` events in 6 weeks of data.** Traced cause: a vessel undocks
from a berth polygon and the very next fix is already outside *all* polygons
of that terminal (because there is no approach polygon to keep the envelope
open). `_step_open_ocean` then emits `zone_exit` immediately and the
MOORED → DEPARTED watchdog never gets two qualifying observations to fire.
Verified against MMSI 256361000 at Sabine on 2026-05-15: vessel undocked at
21:23, sog crossed 1.0 at 21:25, `zone_exit` fired at 21:29 — but no
`departed`, because by 21:29 the fix was already outside the small Sabine
berth+anchorage union.

**Direct downstream consequence:** `pipeline/signal.py` (planned in README)
intends to aggregate `laden ton-miles in transit` from per-MMSI `departed →
next zone_entry` legs. With zero `departed` events that plan produces zero
legs. The signal as designed cannot work until approach polygons exist.

### 1.2 `pipeline/signal.py`, `data/eia.py`, and `analysis/` are still unimplemented

These are the *actual project deliverables*. The infrastructure beneath them
is largely complete; the signal layer is not.

### 1.3 FSRU host MMSIs are missing for 6 of 8 FSRU terminals

`terminals.fsru_host_mmsi` populated only for Brunsbüttel, Eemshaven,
Wilhelmshaven 1 & 2. Missing: **Klaipeda, Mukran, Alexandroupolis, Krk,
Piombino, Ravenna**. The FSRU short-circuit in `pipeline/port_events.py:373-438`
skips them with a warning, so those terminals contribute *zero* port events
from their resident regasification vessel.

### 1.4 Several oversized polygons (likely manual-drawing errors)

```
Alexandroupolis FSRU · berth   · 46.39 km²    (should be ~0.1)
Adriatic LNG         · berth   · 20.77 km²    (should be ~0.1)
Ravenna FSRU         · berth   ·  2.36 km²
```

A 46 km² "berth" will match *every* fix in the local approach to
Alexandroupolis and force a permanent `moored` resolution. Probably an
anchorage polygon mislabelled or drawn at the wrong layer.

### 1.5 Major terminals lack anchorage polygons

`Dunkerque, South Hook, Cove Point, Elba Island, Adriatic LNG, Ravenna FSRU,
Alexandroupolis FSRU` — no anchorage polygon. No queue-time signal computable
at these terminals (and queue time is one of the headline metrics the
`anchorage_entry/exit` events were just added for).

## 2 · Data quality observations

### 2.1 Ingest is healthy and fast

- 20.9 M `ais_fixes` over 6 weeks · 3.4 M `vessel_state` · median end-to-end
  lag **1.07 s**, p95 1.22 s.
- 4910 distinct MMSIs, 157 in-scope (LNG carriers + FSRUs).

### 2.2 Enrichment status looks correct

```
ok        1058    no_imo  1679    error  8    not_found  8    pending  2157
```

The 2157 "pending" are sub-IMO / non-tanker vessels that drift through the
bounding boxes — expected behaviour of zone-gated enrichment.

### 2.3 Zone activity is heavily unbalanced

Last 14 days, fixes-inside-terminal-polygons for in-scope vessels:

```
nweurope    22 626      usgulf  14 927
wmed           55       iberian     27       usatlantic     17
baltic          0       emed         0
```

`nweurope` + `usgulf` are doing all the work. `baltic` and `emed` contribute
**zero** events. Some of this is genuine market reality (low LNG arrivals at
Mukran / Swinoujscie this period), but the missing FSRU host MMSIs (1.3) and
missing / oversized polygons (1.4, 1.5) suppress events that should exist.

### 2.4 Laden inference works but has 14 % NULL rate

```
draught          976  events       flow_direction   98       NULL  171
```

171 / 1245 = 14 % undecidable. These are mostly anchorage-only visits
(`side='no_moored'`), which `infer_laden` correctly returns NULL for — by
design. Worth confirming you don't expect to derive laden state for queue-only
events; if you do, the easiest fix is to forward-fill the most recent draught
and accept the staleness.

### 2.5 `ais_fixes.source` and `vessel_state.source` only ever hold `'aisstream'`

Zero rows have `source='vesselfinder'`. The README mentions "weekly
reconciliation from VesselFinder," but no ingestion code path writes fixes /
state from VF. The discriminator column is dead — either implement the
reconciliation or drop the column.

## 3 · Code quality / correctness gaps

### 3.1 `vf_api_key` defaults to `""` (`config.py:12`)

Silent default; missing key produces opaque 401s from VesselFinder. Either
make it required or validate non-empty before any enrichment call.

### 3.2 `viz/tui.py` duplicates `Settings` (lines 34-48)

Copy-paste of `config.Settings` rather than `from config import settings`.
Minor, but means env-var changes have to be made twice.
`model_config["extra"]="ignore"` is doing the work of swallowing the AIS key.

### 3.3 `viz/tui.py` hand-duplicates `_ZONES` from `AIS_BOUNDING_BOXES`

Lines 53-62 redefine the zone classification by lat/lon ranges. These can
overlap — `wmed (36–46, -2–15)` and `iberian (36.5–44, -10–-2.5)` are
disjoint by luck, but `wmed` and `emed` are close enough that future edits
could collide. The CASE-based classification in `_ZONE_CASE_SQL` will
misattribute fixes if zones overlap. Drive zone assignment from
`terminals.zone` instead.

### 3.4 `port_events.py:_classify_envelope_sides` has a no-op (line 268)

`close_idx = i if envelope_ends_here else i` — always `i`. Vestigial.

### 3.5 `make port-events` is full recompute only

As `ais_fixes` grows past a few months the TRUNCATE-and-rebuild gets slow.
The state machine's per-vessel design makes incremental recompute tractable
(rebuild only MMSIs with new fixes since last run); worth adding before the
table is huge.

### 3.6 `vessel_registry.exclusion_reason` + `excluded` columns look unused

No code path sets `excluded=TRUE`. If the intent (CLI to manually exclude
bad-data MMSIs) hasn't materialised, drop the columns.

### 3.7 Schema-vs-Alembic duality requires care

`schema.sql` is the source of truth and the Docker init script; Alembic
migrations are recorded but only applied via `alembic stamp head` after
`make reset`. Risk: a future migration that doesn't also get hand-merged into
`schema.sql` will silently diverge for fresh DBs. Consider documenting "edit
`schema.sql` in the same PR as any new migration" or generating one from the
other.

### 3.8 Test coverage is asymmetric

Strong: `state_machine` (~400 lines, 9 named scenarios), `laden` (15 cases).
Missing: any test for `ingestion/aisstream.py` parsing (especially the custom
Go-timestamp validator), `models.py` VesselFinder responses,
`dynamic_enrichment.maybe_queue`, the FSRU short-circuit, the spatial-join
SQL in `port_events.py`. Pure-function coverage is great; boundary /
integration coverage is thin.

### 3.9 No CI

There is `make`, `ruff`, `pytest` — but nothing runs them automatically on
push.

## 5 · What's impressive

- **Pure-functional state machine in `pipeline/state_machine.py`** — DB-free,
  deterministic, generator-based, with elegant three-layer fix resolution
  (berth override → stickiness → cold entry) and inline envelope
  reattribution for shared anchorages (the Sabine / Golden Pass case). The
  DFA validator and the post-walk `reattribute_overlaps` safety net are
  belt-and-braces in a good way. Test coverage is genuinely thoughtful —
  golden path, AIS dropout, cold-start (berth and anchorage), re-anchoring,
  drive-by anchorage, distinct-terminals-real-switch.
- **Layered laden inference (`pipeline/laden.py`)** — the
  draught-with-lookahead-window-for-outbound + `flow_direction` fallback is
  sophisticated and well-tested. The 30-90 min undock-to-rebroadcast lag is a
  real-world quirk handled correctly.
- **Zone-gated dynamic enrichment** — avoids spending VF credits on the
  thousands of non-LNG tankers that drift through the boxes. Smart cost
  design.
- **Ingestion robustness** — connection pool, watchdog with timeout-driven
  reconnect, heartbeat for monitoring, `ON CONFLICT DO NOTHING` for
  idempotence, separate handling for `PositionReport` vs `ShipStaticData`.
  Median 1 s lag, p95 1.2 s.
- **Custom Go-timestamp parser in `models.py`** with the nano-second
  truncation — the kind of detail that would silently corrupt timestamps if
  missed.
- **TUI in `viz/tui.py`** — per-zone sparklines, ingest lag histogram,
  terminal staleness, enrichment status. Slow PostGIS query correctly
  throttled to 30 s while fast counters run every 2 s.
- **TimescaleDB choices** — 1-day chunks, continuous aggregates for
  fixes/minute and fixes/hour, GIST index on `geom`. Right tool, right
  config.
- **Three-layer documentation** (README → CLAUDE.md → inline module
  docstrings) is rare and high-signal. The CLAUDE.md is one of the best
  agent-context files I've seen.

## 6 · Recommended next steps, in order

1. **Draw `approach` polygons** for all 33 in-scope terminals (highest
   leverage; without this, no `departed` events, no signal). Start with the
   US Gulf channel transits (Sabine, Calcasieu, Freeport, Corpus Christi)
   since they have the longest channel and the largest LNG flow.
2. **Add the 6 missing `fsru_host_mmsi` values** (Klaipeda, Mukran,
   Alexandroupolis, Krk, Piombino, Ravenna). A few hours of MarineTraffic
   lookups.
3. **Fix the oversized berth polygons** (Alexandroupolis 46 km², Adriatic
   LNG 21 km², Ravenna 2.4 km²) — these are corrupting resolution at those
   terminals.
4. **Draw missing anchorage polygons** at Dunkerque, South Hook, Cove Point,
   Elba Island, Ravenna FSRU.
5. **Implement `pipeline/signal.py`** — per-MMSI `departed → next zone_entry`
   legs × `ST_Distance(::geography)` × `dwt × laden_flag`, aggregated by
   `(origin_zone, dest_zone)`.
6. **Implement `data/eia.py`** — straight EIA API client; small surface.
7. **Update `qgis/README.md`** to reflect actual layer types (add `approach`)
   and current coverage.
8. **Add CI** running `ruff` + `pytest` on push.
9. **Add an incremental mode for `make port-events`** before the recompute
   time becomes painful.
10. **Decide on the `vesselfinder` source / `excluded` / `exclusion_reason`
    columns**: implement or drop.

---

The bones are excellent. The blocking issue is not architectural — it's that
the `approach` polygon work was left undone, and the state machine has been
silently producing a degenerate event sequence as a result. Fixing that one
thing unblocks 60 % of the remaining roadmap.
