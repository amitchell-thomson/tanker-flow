# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**tanker-flow** derives leading Henry Hub / European LNG price spread signals from LNG carrier positions. It ingests AIS (Automatic Identification System) vessel position data ‚Äî live, plus a decade backfilled from NOAA and Global Fishing Watch ‚Äî processes it into port events at major LNG export/import terminals across seven geographic zones, and aggregates them into a **34-signal daily panel** (`signal_daily`) of leading indicators: gas-in-transit volume, loading/discharge pace, anchorage queues, voyage speed, floating-storage age and outage detection. Every signal is built dual-basis (a leakage-safe `knowable` series for modelling alongside a hindsight `physical` series for validation). The full catalogue is `analysis/SIGNALS.md`.

## Architecture

### Data Flow
```
AISstream WebSocket ‚Üí ingestion/aisstream.py ‚Üí ais_fixes (TimescaleDB hypertable)
VesselFinder API   ‚Üí ingestion/vesselfinder.py ‚Üí vessel_registry (enrichment)
VesselFinder API   ‚Üí ingestion/vf_rescue.py   ‚Üí ais_fixes (live-position backstop for AIS gaps)
IGU orderbook CSV  ‚Üí scripts/discover_newbuilds.py ‚Üí vessel_registry (budgeted newbuild discovery)
ais_fixes          ‚Üí pipeline/port_events.py  ‚Üí port_events (state machine)
port_events        ‚Üí pipeline/legs.py          ‚Üí classified voyage legs (in-memory)
legs + visits + queues ‚Üí pipeline/signal.py     ‚Üí signal_daily (34-signal panel, dual-basis)
EIA API            ‚Üí data/eia.py               ‚Üí US natural gas storage (Henry Hub fundamentals)
```

### Database Schema (`db/init/schema.sql`)
`schema.sql` is the source of truth for the full current schema. It is the Docker init script ‚Äî `make reset` recreates the DB from it. Alembic migrations in `db/migrations/` record how the schema evolved; after a fresh `make reset`, run `alembic stamp head` so Alembic treats the schema as already applied before adding future migrations.

Tables and views:

- **`ais_fixes`** ‚Äî TimescaleDB hypertable partitioned by `fix_ts` (1-day chunks). Append-only raw fixes from two sources: `aisstream` (real-time WebSocket, labelled `aisstream-mmsi-{1,2,3}` per connection) and `vesselfinder` (weekly reconciliation **and** the `vf_rescue.py` live-position backstop ‚Äî both inject as normal fixes). The `source` column distinguishes them.
- **`vessel_state`** ‚Äî TimescaleDB hypertable. Voyage-specific fields (draught, destination, ETA) from `ShipStaticData` messages, keyed by `(state_ts, mmsi)`.
- **`vessel_registry`** ‚Äî One row per MMSI. Populated passively as vessels are seen; enriched with VesselFinder masterdata. `enriched_at IS NULL` means not yet enriched. `vf_enrichment_status` tracks enrichment outcome (`ok` | `not_found` | `error` | `no_imo`). Enrichment is zone-gated (queued only when a fix lands inside a `terminal_zones` polygon), so pending rows outside terminals are noise rather than a backlog to clear.
- **`terminals`** ‚Äî One row per LNG terminal. Has a `zone` column grouping terminals into seven geographic zones (see below). `in_signal_scope` flags whether the terminal is active in the signal. `is_fsru` distinguishes FSRUs from land-based terminals. `unlocode` (UN/LOCODE like `NLRTM`) is seeded via `make seed-unlocodes` and used by `pipeline/dest_parser.py` to resolve `vessel_state.dest` ‚Üí `terminal_id` for tier-2 scoring.
- **`terminal_zones`** ‚Äî PostGIS polygons imported from QGIS. `zone_type` is one of `berth` | `anchorage` | `approach`. `approach` is a macro-envelope drawn to geometrically contain anchorage + channel + berth so a single visit stays "open" while the vessel transits between them (otherwise the ~4‚Äì6 h channel transit at Sabine/Rotterdam etc. would emit a spurious zone_exit). One or more rows per terminal, distinguished by `sub_zone`.
- **`port_events`** ‚Äî Derived table, recomputed by `make port-events` from `ais_fixes` + `terminal_zones`. One row per per-vessel transition. `event_type` in {`zone_entry`, `anchorage_entry`, `anchored`, `anchorage_exit`, `moored`, `departed`, `zone_exit`}. `anchorage_entry`/`exit` are raw polygon-crossing markers (no dwell/SOG filter) ‚Äî they bracket every visit to an anchorage and let downstream queue-time signals be measured without the ~30 min back-dating bias of `anchored`. `anchored` remains the dwell-confirmed marker (‚â•30 min stationary at sog<1). `zone` is one of seven geographic zones; `terminal_id` names the specific terminal. `lat`/`lon` carry the event position (used downstream for great-circle distance). `regime` is a stored generated column tagging ingestion fidelity (`noaa`/`gfw`/`bbox`/`mmsi_filter`). `cold_start = TRUE` flags synthetic events emitted when a vessel's first observed fix was already inside a polygon. `laden_flag` is derived per event by forward-filling the most recent `vessel_state.draught` and comparing to `vessel_registry.design_draught` (‚â•0.85 √ó design ‚áí laden).
- **`signal_daily`** ‚Äî Derived table, rebuilt by `make signals` (TRUNCATE + rebuild, like `port_events`) from the three port-event pairings: voyage legs (`pipeline/legs.py`), port visits (`pipeline/visits.py`), anchorage queues (`pipeline/queues.py`). Tidy/long daily panel: one row per `(signal_key, bucket_date, zone_scope, regime, basis)` carrying `value`, `n_legs`, and three confidence columns (`value_dispersion` = MAD of distributional signals; `open_fraction` = censoring exposure; `estimated_fraction` = reliance on an estimated magnitude). **34 signal keys** across six families ‚Äî headline gas-volume (`gas_loading_us`/`gas_discharging_eu`/`gas_in_transit_volume`/`gas_ballast_to_us`), export pace, import absorption, floating-storage/voyage urgency, arbitrage/flow geography, and fleet/shocks; full definitions in `analysis/SIGNALS.md`. `regime` is segmented per SIGNALS.md ¬ß2.1 (never aggregate across the 2026-05-30 seam; the synthetic `'all'` row double-counts US signals ‚Äî use `regime='noaa'` there). **Both bases are built every rebuild:** `basis='physical'` (hindsight-clean, validation only) and `basis='knowable'` (leakage-free point-in-time, the model input ‚Äî `MODELS.md` consumes it only). `signal_daily_live_vintage` is an append-only log of the live values as-printed each day, backing the knowable self-validation.
- **`tier_promotions`** ‚Äî Append-only log of vessels promoted **up into** the persistent band (tiers 1-3), written by `scoring.py` (`via='scoring'`) and the inline state machine (`via='inline'`). Powers the TUI promotions panel.
- **`vf_rescue_log`** ‚Äî Append-only audit trail **and** restart-safe credit ledger for `vf_rescue.py`. One row per VF live-position lookup attempt: `rescue_class`, `result` (`rescued` | `no_position` | `rejected_stale` | `rejected_teleport` | `error` | `dry_run` | `skipped_budget`), `credits` billed, and `recheck_at` (per-vessel cooldown). Today's `SUM(credits)` gates the daily cap; the latest row per MMSI is the cooldown. `skipped_budget` rows (0 credits, no cooldown, one per vessel per UTC day) audit candidates the daily budget couldn't serve ‚Äî a vessel with a skipped row and no later billed row the same day is measured unmet demand. **`scripts/discover_newbuilds.py` also logs here** (`rescue_class='discovery'`, `result='rescued'`) so newbuild-discovery catches share this credit ledger with rescue ‚Äî today's `SUM(credits)` therefore reconciles both workers against the glide budget.
- **`vf_account_status`** ‚Äî VesselFinder account-balance snapshots from the free `/status` endpoint, appended each rescue run. Latest row = current `credits` + `expiration_date`; consecutive rows give the true burn rate.
- **`ingestion_events`** ‚Äî Append-only lifecycle log written by `aisstream.py` (connect / subscribed / planned_reconnect / watchdog_reconnect / disconnect / error). Per-connection liveness is derived from `ingestion_stats_minute` (max bucket per source) rather than a separate heartbeat table.
- **`ingestion_stats_minute` / `ingestion_zone_minute`** ‚Äî Per-minute ingestion stats (lag mean/p95, distinct_mmsi, queue saturation, connection age; per-zone fix counts in the latter) written by the in-process `MinuteAggregator`. Drive the TUI health and field zones.
- **`priority_watchlist`** ‚Äî Derived hourly by `pipeline/scoring.py`. One row per LNG/FSRU vessel in scope (~780 today), ranked into 5 tiers. The ingester reads `top 100 WHERE tier<=3` (open-leg + in-port pins first) for persistent slots, and for scan rotation reads 50 from four priority-ordered pools: **persistent-band overflow** (tier‚â§3 that missed a persistent slot ‚Äî the closest such vessels held the slots, the rest rotate here), then tier-4, then a reserved tier-5 discovery quota, then a reserved **FSRU host-watch quota** (`SCAN_OVERFLOW_SLOTS`/`SCAN_TIER4_SLOTS`/`SCAN_TIER5_SLOTS`/`SCAN_FSRU_SLOTS` = 15/22/10/3, with roll-over). The overflow pool closes the old coverage hole where tier-3 vessels crowded out of the persistent block were excluded from the tier‚â•4-only scan and went fully dark. **FSRUs are force-demoted to tier 5 (`scoring.FSRU_TIER`) and excluded from every other pool**: a deployed FSRU sits moored at its host for months and its own AIS fixes never drive the signal (see FSRU short-circuit below), so it earns no persistent slot ‚Äî the small dedicated quota gives the ~46 FSRUs only an occasional relocation check. They're also excluded from inline in-zone promotion and the in-port pin so nothing bounces them back into the persistent band. `in_slot` / `slot_kind` are written by `aisstream.py` after each 1h reconnect so the TUI can render who's currently subscribed; `last_scan_window_at` is bumped to `now()` inside `load_scan_mmsis` itself so the same vessels don't get re-picked on every watchdog reconnect.
- **`fixes_per_minute` / `fixes_per_hour`** ‚Äî TimescaleDB continuous aggregates over `ais_fixes`, used by the monitoring TUI.

### Ingestion (`ingestion/`)
- **`aisstream.py`** ‚Äî Async WebSocket subscriber using `websockets` + `asyncpg`. Runs three parallel WebSocket connections, each subscribed to a disjoint chunk (‚â§50 MMSIs per AISstream's `FiltersShipMMSI` cap). Slot allocation: **chunks 0+1 are the persistent block** (top 100 vessels by tier from `priority_watchlist`); **chunk 2 is the scan rotation** (50 slots drawn from four priority-ordered pools ‚Äî persistent-band overflow ‚Üí tier-4 ‚Üí tier-5 ‚Üí FSRU host-watch, with roll-over; see `priority_watchlist` above ‚Äî each ordered by `last_scan_window_at ASC NULLS FIRST`; the loader writes back `now()` in the same transaction so each reconnect ‚Äî planned 1h *or* the 5-min watchdog ‚Äî picks a different 50). Each subscription uses a full-globe bbox; the MMSI filter does the constraining. Source labels (now plumbed through to `ais_fixes.source` as well as the stats tables): `aisstream-mmsi-1` / `aisstream-mmsi-2` / `aisstream-mmsi-3`. `RECONNECT_INTERVAL_SECONDS = 3600` triggers a planned reconnect every 1h, and a `scoring_loop` background task re-runs `pipeline.scoring.compute_and_upsert(pool)` just before each one so the priority_watchlist is fresh.
- **`dynamic_enrichment.py`** ‚Äî Worker draining the VesselFinder enrichment queue (rate-limited to 1 req/s). The old passive-discovery hook (`maybe_queue()` triggered by unknown-MMSI fixes in terminal polygons) has been removed: under server-side MMSI filtering, unknown MMSIs never reach us. The queue is now fed by `vesselfinder.py --terminal-only` (batch path) and the daily newbuild-discovery worker (`scripts/discover_newbuilds.py`).
- **`vesselfinder.py`** ‚Äî One-shot batch enrichment via the VesselFinder masterdata API. Defaults to `--terminal-only` (only MMSIs with at least one fix inside a `terminal_zones` polygon). Rate-limited to 1 req/s. Vessels with `imo = 0` are sub-IMO and short-circuited to `vf_enrichment_status = 'no_imo'` without an API call (LNG carriers always have IMOs from day one, so the `imo=0 ‚Üí real IMO` transition is intentionally not handled).
- **`vf_rescue.py`** ‚Äî Live-position backstop for AIS gaps, run as a background task in `aisstream.py` (every 30 min) and via `make vf-rescue`. The signal is built from leg-defining port *events* (laden `departed` from an export terminal ‚Üí `zone_entry`/`moored` at an import terminal), so when AISstream drops a vessel **at or approaching one of our terminals** that event is at risk. The worker selects coastal vessels gone AIS-silent within an actionable band (proximity + heading gates; a `STALE_CEILING_HOURS` ceiling skips vessels whose event has already passed ‚Äî except open visits (moored/anchored, no `departed` yet), which keep the higher `OPEN_VISIT_STALE_CEILING_HOURS` since a late fix still closes the visit and opens the leg), fetches their current position from VF's `/vessels` feed (terrestrial ‚Äî 1 credit; the target is coastal so satellite is never requested), sanity-checks it (freshness + teleport gates), and injects it as a normal `ais_fixes` row (`source='vesselfinder'`) ‚Äî the existing pipeline re-acquires it for free. Candidates are classified by the event at risk (`import_arrival` / `export_departure` / `outage_check` highest, then `dest_capture` / `import_berth`, then `export_arrival` / `eta_arrival`; `manual` via `--mmsi` jumps the queue). `export_arrival` (ballast approaching to load) is a low-value class ‚Äî it only times the ballast-leg *close* (`gas_ballast_to_us`), never a headline laden signal ‚Äî so it's held to a stricter bar than the laden classes: it fires only in final approach (‚â§`FINAL_APPROACH_KM`, or actively closing) **and** after ‚â•`EXPORT_ARRIVAL_MIN_SILENCE_HOURS` (8h, 2√ó the general floor), trimming its loiter/short-gap tail; reacquisition still closes the leg from the next live fix. `eta_arrival` is the speculative discovery class (#7): a silent carrier with an *imminent parsed ETA* (`ETA_RESCUE_HORIZON_HOURS` ahead / `ETA_RESCUE_PAST_GRACE_HOURS` past) gets one coastal poll even with no near-terminal fix ‚Äî the missed-arrival case (CLEAN VITALITY, 2026-06-08) where a FOR-ORDERS vessel never earned a slot and went dark in tier 5; capped at `ETA_RESCUE_MAX_VESSELS`/run, surplus-only, so it depends on the budget being ahead of the glide line. VF credits are a finite reserve that **expires unused** on a fixed date, and every credit spent on a near-terminal silent vessel buys signal, so the policy is to deplete the reserve to ~zero *exactly at expiry* (spend slower ‚áí forfeit credits; faster ‚áí go dark for the final stretch), not to minimise it: the daily cap is **derived each run** from the latest `vf_account_status` snapshot (`glide_cap`: ceil(reserve √∑ days-to-expiry), ‚âà **14/day** as of 2026-06; `DAILY_CREDIT_CAP` is the no-snapshot fallback) and the budget is **split by priority** (`split_budget`): priority-0 classes (the leg-defining events + `manual`) are exempt from the glide cap ‚Äî bounded only by the `GLIDE_CAP_CEILING` total-daily brake ‚Äî while priority-‚â•1 classes spend only the **surplus vs the glide line** (`glide_surplus`: live balance minus the straight line from the anchor snapshot to zero-at-expiry), so P0 overspend automatically starves P‚â•1 until the line recovers (leaky-bucket payback, no persisted state); candidates the budget can't serve are logged as `result='skipped_budget'` so unmet demand is measurable. `vf_rescue_log` is the restart-safe daily-credit ledger + per-vessel cooldown; a `no_position` result backs off exponentially per consecutive miss (12h ‚Üí 24h ‚Üí ‚Ä¶ ‚Üí 7d, reset by any returned position) so terrestrially invisible vessels stop eating plan slots. `--dry-run` previews candidates/cost without spending; `--status` snapshots the free account balance.
- **`scripts/discover_newbuilds.py`** ‚Äî Budgeted daily newbuild discovery; run via `make discover` (`make discover-dry` to preview) on a daily schedule. Under the closed-loop MMSI filter there is no passive discovery, so the **only** way a delivered LNG carrier enters the system is IGU orderbook ‚Üí VF resolves IMO‚ÜíMMSI ‚Üí `vessel_registry` ‚Üí subscription. The worker sweeps orderbook hulls with `delivery_year ‚â§ current year` not yet in the registry and resolves the ones VF can now map to a live MMSI (reusing `import_igu_fleet`'s VF VESSELS + master ‚Üí registry/fix/state path). **VF bills per returned record**, so an undelivered hull (404/empty) costs **0 credits** ‚Äî a full sweep of unresolved hulls is free, and only a genuine catch costs the 3-credit master+AIS lookup. The budget is therefore a **brake, not a throttle**, and shares the rescue reserve: `discovery_credit_budget` (in `vf_rescue.py`, beside `split_budget`/`glide_surplus`) takes the better of a small guaranteed **floor** (one catch/day even when rescue has eaten the whole glide, so deliveries are never starved) and the **glide surplus** (faster on slack days), capped by a rareness ceiling and the shared `GLIDE_CAP_CEILING` disaster brake. Real cost is ~90cr/yr (self-limiting: the candidate pool drains as hulls are caught). Catches log to `vf_rescue_log` (`rescue_class='discovery'`) and a post-run `/status` snapshot lets the next rescue pass see the spend. The candidate **pool** itself is widened only by the annual `make refresh-fleet` (a new IGU PDF turns "Hull NNN" orderbook rows into named/resolvable vessels).
- **`models.py`** ‚Äî Pydantic models validating raw AISstream JSON. Key quirk: `MetaData.time_utc` arrives as a non-standard Go timestamp string (`"2026-04-12 19:10:08.192247737 +0000 UTC"`) and is parsed by a `field_validator`. `MaximumStaticDraught` of `0.0` is treated as `NULL` (unreported). Also holds the VesselFinder masterdata models and `VesselFinderAIS` (the live `/vessels` position parsed by `vf_rescue.py`, with the same sentinel-nulling: COURSE 360 / HEADING 511 / DRAUGHT 0 ‚Üí NULL).

### Pipeline (`pipeline/`)
- **`port_events.py`** ‚Äî Orchestration. TRUNCATEs `port_events`, runs a single on-the-fly spatial join via `ST_Within` to attach a candidate-zones array to every in-scope fix, streams the result via asyncpg cursor split by MMSI, feeds each vessel's stream into the pure state machine, post-processes (laden, DFA validation), and batch-inserts the events. In-scope vessels are `vessel_registry WHERE is_lng_carrier = TRUE OR is_fsru = TRUE` ‚Äî FSRUs are admitted explicitly because VesselFinder classifies them as `'Offshore Support Vessel'` (so `is_lng_carrier = FALSE`).
- **`scoring.py`** ‚Äî Builds the `priority_watchlist` table by ranking every LNG/FSRU vessel into one of 5 tiers from current `ais_fixes` + `vessel_state` + spatial joins against `terminal_zones` and the `config.ZONES` rectangles. Tiers: 1 = in any terminal_zones polygon (last **3d** ‚Äî tightened from 14d so we don't waste persistent slots on vessels long-since departed); 2 = a parsed ETA within `ETA_IMMINENT_HOURS` ahead (or `ETA_PAST_GRACE_HOURS` just past, so a slightly-late vessel stays pinned through berthing) ‚Äî **not** gated on a resolved dest (a "FOR ORDERS" carrier with an imminent ETA still promotes; rescues long-voyage vessels whose declaration is stale/absent but arrival is imminent, e.g. ballast carriers approaching a US load terminal that broadcast no parseable dest) **OR** declared inbound via parsed `vessel_state.dest` to a known terminal with state_ts < 14d; 3 = in the wider config.ZONES bbox (14d), ordered within-tier by **closing-ness** (proximity + heading to the nearest zone via `_closing_bonus`) so the scarce slots go to vessels actually approaching; 4 = any fix in last 7d; 5 = stale/unseen. **FSRUs short-circuit all of this** ‚Äî they're force-assigned `FSRU_TIER` (=5) regardless of position (a deployed FSRU would otherwise score tier 1 forever and waste a persistent slot for no signal benefit) and routed to a dedicated low-frequency scan quota (see `priority_watchlist`); they're also dropped from the in-port pin here and from inline promotion in `aisstream.py`. Ingester reads top-100 by `(tier ASC, score DESC)` for persistent slots; scan rotation covers tier-3 persistent-band overflow + tiers 4/5 + the FSRU quota (see `priority_watchlist` above). `is_pinned` covers two cases: an **open leg in its expected approach window** ‚Äî both directions, laden‚Üíimport arrival *and* ballast‚Üíexport-terminal loading (the ballast return was the dominant appear-in-berth miss), gated by `EXPECTED_VOYAGE_DAYS` per departure zone (US export ‚Üî EU import ~14‚Äì18d) and ranked by closeness to expected arrival so the scarce slots favour vessels approaching *now* rather than just-departed ones still mid-ocean (these are coarse Phase-1 constants; Phase 2 swaps them for per-O-D rolling medians from `legs.py` durations) ‚Äî **and** a vessel currently open in a port visit (in-port pin ‚Äî protects long berth queues from decaying out of coverage). Single sub-second SQL pass + in-memory tier-assignment loop. Runs once at ingester startup, then every 1h as a background task; also runnable manually via `make scoring`.
- **`dest_parser.py`** ‚Äî Pure function `parse_destination(dest_str, unlocode_to_terminal) -> (terminal_id, is_for_orders)`. Handles UN/LOCODE direct (`NLRTM`), internal-space LOCODE (`NL RTM`), chained (`USSAB>NLRTM` ‚Üí uses RHS), "FOR ORDERS" markers, and freeform names (`ROTTERDAM`, `EEMSHAVEN`) via an in-code normalizer. Tier-2 scoring relies on this. Aliases for known LNG terminals live as constants in this module; UN/LOCODEs themselves live on `terminals.unlocode`.
- **`state_machine.py`** ‚Äî Pure per-vessel walk; no DB. States: TRANSIT / IN_ENVELOPE / ANCHORED / MOORED / DEPARTED. **Three-layer fix resolution**: (1) berth override ‚Äî any candidate at `zone_type='berth'` wins regardless of stickiness; (2) stickiness ‚Äî if a visit envelope is open for terminal A and any candidate matches A, stay with A; (3) cold entry ‚Äî most-specific `zone_type` wins (berth > anchorage > approach), tiebreak by nearest berth centroid (over all sub_zones). **Inline reattribution**: when a berth-override would force a terminal switch mid-envelope, if every earlier event in the current envelope had the new terminal in its `candidate_terminal_ids`, the envelope is rewritten in place instead of emitting spurious `zone_exit`/`zone_entry`. **Anchorage tracking**: while in IN_ENVELOPE or ANCHORED, every fix's resolved `zone_type` is compared to an `in_anchorage` flag ‚Äî entering the anchorage polygon emits `anchorage_entry`, leaving emits `anchorage_exit`. These are suppressed while MOORED/DEPARTED to avoid spurious events from jitter near overlapping berth/anchorage polygons. `anchorage_exit` also transitions ANCHORED ‚Üí IN_ENVELOPE, allowing re-anchoring in the same envelope. **Cold-start**: a first-observed fix already inside a berth/anchorage emits synthetic `zone_entry` (+ `anchorage_entry` for the anchorage case) + `moored`/`anchored` with `cold_start=TRUE`. **AIS-dropout safe**: transitions require an actual fix *outside* the relevant polygon; time gaps alone don't trigger anything. **Back-dated timestamps** for dwell-confirmed events (`anchored`, `moored`, `departed`): they fire at the moment of transition (first qualifying fix), not at dwell-confirmation. Raw polygon crossings (`anchorage_entry`/`exit`, `zone_entry`/`exit`) fire at the actual observed fix. **Reacquisition after a gap**: a berth fix (sog < 1) that is the first of its arming sequence *and* lands ‚â• the moored dwell window after the previous fix confirms `moored` immediately (back-dated to that fix) instead of waiting out the dwell ‚Äî a single isolated fix can never exhibit the 30-min dwell, so this captures vessels picked back up already alongside (e.g. a one-off `vf_rescue.py` fix after a mid-visit AIS dropout), same reasoning as cold-start.
- **`laden.py`** ‚Äî Builds an in-memory `mmsi ‚Üí [(state_ts, draught)]` lookup from `vessel_state` once per run, then bisect-lookups the most recent draught at-or-before each event's time. Forward-fill (any age) because `vessel_state` is sparse.
- **`legs.py`** ‚Äî Voyage-leg foundation under the flow signals. Pure `pair_legs` + thin `compute_legs(pool, now)` loader. Pairs each `departed` with its vessel's next `zone_entry` and **classifies**: `closed` (cross-zone real voyage), `same_zone` (intra-region hop ‚Äî ~zero cross-zone distance, excluded from lane signals), `open_in_transit` (no arrival yet, within the per-O-D voyage window), `open_floating` (past window but a recent coastal fix ‚Üí genuine floating storage), `open_arrival_gap` (past window, stale fix in dest region ‚Üí arrived-and-missed), `open_censored` (past window, no coastal evidence ‚Üí phantom, excluded everywhere). Each leg is regime-tagged (`config.regime_of` on the departure) and carries `dwt`/`gas_capacity_m3` + great-circle `distance_nm` (closed legs only). The signal layer aggregates over this; it does **not** re-pair.
- **`visits.py`** ‚Äî Port-visit pairing: pure `pair_visits` + thin `compute_visits(pool, now)`. A *visit* pairs a `moored` with the vessel's next `departed` ‚Äî the berth-occupancy interval during which a vessel loads (export terminal) or discharges (import terminal). Carries terminal/zone/`flow_direction` + `dwt`/`gas_capacity_m3` + the moored event's `laden`/`regime`/`cold_start`. An **open** visit (a `moored` with no following `departed`) is the vessel currently in berth; `departed_ts=None`. This is the foundation under `gas_loading_us` / `gas_discharging_eu` and the berth-turn-time signals; it is distinct from the `anchorage_entry ‚Üí moored` *queue* pairing (`queues.py`, below).
- **`queues.py`** ‚Äî Anchorage-queue pairing: pure `pair_queues` + thin `compute_queues(pool, now)`. The third pairing alongside legs/visits. A *queue* pairs the **first** `anchorage_entry` of a run with the vessel's next `moored` **at the same terminal** ‚Äî the wait before berthing. Robust to the exit/re-enter jitter as a vessel leaves the anchorage for its berth (brackets first-entry‚Üímoored, absorbs intermediate crossings); the same-terminal guard rejects pairing a US anchorage with the EU mooring 16 days later (a whole voyage). Carries `queue_h`, `anchored_seen` (real wait vs drive-by clip ‚Üí #15 vs #16), and a secondary `anchorage_dwell_h` (first-entry‚Üílast-exit) diagnostic. Foundation under `load_queue_h`/`discharge_queue_h` (#6/#12), the queue-depth (#7/#13) and queue-rate (#15/#16) signals. EU queues are live-only (GFW carries no anchorage events); US queues reconstruct to 2016 from NOAA.
- **`signal.py`** ‚Äî The signal aggregation layer. Pure aggregators (`accumulate_daily`) + a thin DB loader; rebuilds `signal_daily` (TRUNCATE + atomic executemany swap, like `port_events`). Consumes `legs.compute_legs()` + `visits.compute_visits()`. Headline signals are **gas volumes (m¬≥)** reconstructed per day: the at-sea legs as a **stock** over each leg's live interval (`gas_in_transit_volume`, `gas_ballast_to_us`, banded by destination zone), and the berth visits as an **amortized flow** (`gas_loading_us` / `gas_discharging_eu`, banded by terminal). The flow spreads each cargo across its berth hours at a constant rate (`amortized_cargo_contribution`) so a closed visit integrates to exactly one cargo (m¬≥/day); an **open** visit estimates its total dwell from the terminal's closed-visit mean (`terminal_dwell_hours`) until it departs, with the cumulative deposit capped at one cargo. The unit is `gas_capacity_m3` ‚Äî **no distance weighting**. Beyond the four gas-volume headlines, `signal.py` also builds the **event/measurement signals** via `accumulate_events` (median+MAD / count / fraction for berth turn-time, voyage speed/anomaly/slow-steam, loadings, round-trip, queue time/rates), the **fleet stocks** (`fleet_daily`), and the **outage/anomaly** signals (`days_since_*` recency, queue-formation WoW, cold-start rate, newbuilds), plus the intent (declared_eu_share) and composite (net_export_pressure‚Ä¶spread_thrust) families ‚Äî 34 keys total. Phantom guards: `OPEN_VISIT_CEILING_DAYS = 5` (open visits), the open-leg censor in `legs.py`, and `QUEUE_OPEN_CEILING_DAYS = 14` (open queues). **Both bases are built**: `physical` (hindsight, validation) and `knowable` (point-in-time via `knowable_leg_interval` / `amortized_cargo_knowable` / estimated open-queue waits ‚Äî the model input). Distance for the speed signals falls back to terminal-zone centroids when event fixes lack coordinates (every GFW leg). A live rebuild (no `--as-of`) snapshots `signal_daily_live_vintage`. `--as-of`/`--panel-start` for reproducible rebuilds. See `analysis/SIGNALS.md`.
- **FSRU short-circuit** ‚Äî `is_fsru=TRUE` MMSIs skip the state-machine walk; one `moored` event is emitted at the FSRU's declared host terminal (`terminals.fsru_host_mmsi`), timestamped at the FSRU's first observed fix, with `cold_start=TRUE`. FSRUs without a declared host are logged and skipped.

### Data (`data/`)
- **`eia.py`** ‚Äî EIA v2 API loader (one-shot, idempotent batch ‚Äî sibling to `ingestion/vesselfinder.py`). Upserts tidy rows into `eia_series` keyed by `(series_id, period)` `ON CONFLICT DO UPDATE` (EIA revises recent periods). Pure `parse_eia_response` + `merge_rows` (DB-free, unit-tested) + thin async `fetch_and_upsert` (paginates the v2 `/data` endpoint, incremental from `max(period)` with a per-frequency trailing re-pull window for revisions; `--full` backfills). A `SERIES` registry (route + `series` facet + frequency + unit + phase) makes a new series one entry, not new code. **Phase 1 (active):** monthly US LNG exports (`lng_exports`) ‚Äî the capture-rate ground truth (park-checkup #13). **Phase 2 (deferred, `phase=2` in registry, not fetched by default):** weekly Lower-48 storage + Henry Hub daily spot, for the spread-model control set. `make eia` (incremental) / `make eia-full` (backfill); `--probe KEY` live-fetches one page without writing (the step-0 route-verification tool ‚Äî the v2 routes/facets are best-known from legacy v1 IDs and need live confirmation). Empty `EIA_API_KEY` ‚áí "disabled" log, no crash.
- **`capture_rate.py`** ‚Äî Phase 1b: the read-only capture-rate validator (park-checkup #13's long pole). Per month, `captured` (laden `departed` events from US export terminals) √∑ `implied` (EIA monthly export MMcf √∑ mean cargo size) = the fraction of real US LNG exports our AIS pipeline sees. Pure conversion fns (`m3_lng_to_mmcf`/`implied_cargoes`/`capture_rate`) + a pure month-join (`build_rows`), both unit-tested. The m¬≥ LNG ‚Üí MMcf gas constant is a single named, locked value (600√ó volumetric expansion ‚áí ~3.69 Bcf per 174k m¬≥; the design's looser "~3.4 Bcf" is *not* used). A month is **meaningful** only when wholly post-cutover **and** EIA-published **and** revised ‚Äî so it correctly reports "no meaningful month yet" until June 2026 firms (~late summer; the `revised` gate keys off elapsed months, not a hardcoded date). Reports both nominal-174k and observed-mean denominators. No table/writes. `make capture-rate`.
